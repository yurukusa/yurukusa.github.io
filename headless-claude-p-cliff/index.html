1 · Why this page exists

On 2026-05-23, issue #61704 documented a €84.68 unexpected charge. The user asked Claude Code across ~10 separate sessions whether claude -p invocations bill against subscription quota or account balance, received confident affirmations that it was subscription-only, built a 122-session bot pool on that assumption, and discovered the actual routing only when the charge landed against the account balance.

The case is the canonical worked instance for a structural failure mode this page is built to pre-empt: the model cannot read Anthropic's billing backend at inference time, so any claim it makes about which cost bucket -p sessions fall into is training-derived speculation, presented with the same confident framing as any other model output. There is no in-session verification step that compares the claim against the actual routing logic.

This page is for operators in one of these categories:

  • Bot-pool operators — running many concurrent claude -p invocations (the jimpreuss-class case).
  • CI/CD integrators — invoking claude -p from GitHub Actions, GitLab CI, Jenkins, or similar.
  • Scheduled-job operators — running claude -p from cron, systemd timers, or similar.
  • Autonomous-loop operators — running claude -p in a self-driving loop without per-invocation human review.

If you fit one of these categories and have not personally verified the routing against the billing dashboard within the last 30 days, you are exposed to the same failure mode as #61704. The protocol below closes that exposure in 15 minutes of operator time.

2 · What changes on June 15

Anthropic is splitting the Agent SDK billing path starting 2026-06-15. The exact shape, summarized:

Pre-cliff (today) Post-cliff (2026-06-15+)
Subscription Max Plan users running claude -p drew from the same pool as interactive sessions. Agent SDK path (including claude -p) draws from a separate monthly dollar credit, sized by plan tier.
API-key users billed at standard API rates per invocation. API-key users continue at standard API rates (no change).
Mixed-auth scenarios (OAuth → Max Plan; API key → balance) often opaque from the model's perspective. Same opacity, plus the new monthly-credit ceiling for the SDK path on subscription accounts.

The official Anthropic reference for the change is support article 15036540.

Why the cliff matters specifically for automation operators: the credit ceiling on the new SDK path is finite. An automation pattern that consumed unlimited subscription quota pre-cliff may exhaust the new monthly credit, and then either bill at API rates (if extra-usage is enabled) or stop (if it is not). The €84.68-class failure mode after June 15 has a higher ceiling because the extra-usage rates apply once the SDK credit is exhausted, and the model's pre-cliff training data does not reflect the new routing.

3 · Why the model gets billing claims wrong

This is not a bug in any specific Claude model version. It is a structural failure mode in production AI coding agents — what the cluster catalog at #60226 calls recognition-without-arrest: the model articulates a rule it would otherwise apply ("I cannot verify Anthropic's internal billing routing from here") and then proceeds to articulate a confident claim about that exact unverifiable topic.

The mechanism, in three lines:

  1. The claim derives from training corpus statistics. Forum posts, docs, blog entries that the training corpus saw all describe pre-2026-06-15 billing. The model interpolates a "most plausible answer" from that distribution.
  2. The actual behavior derives from Anthropic's billing backend. The backend's routing logic lives outside the model's runtime. The model has no inference-time mechanism to read it.
  3. The verification step would have to compare claim vs actual. That comparison does not exist in the model's runtime. The claim therefore lands with no integrity check on accuracy.

This is the same shape that surfaces at other axes in the cluster: PreToolUse-Bash (scope-claim from operator prompt vs argv from harness), dispatch-end (dispatch-claim from assistant narrative vs actual Agent/Task call), PostToolUse-Edit/Write (write-success-claim from in-memory cache vs disk state). The billing axis is row N+1 in the catalog. The structural mechanism is identical; the substrate (training-derived claim vs backend routing) is the new axis.

The 2026-06-15 reclassification multiplies this failure mode by ~10×. Most existing forum/doc/blog content the model was trained on describes pre-2026-06-15 billing. After June 15, any model claim about -p billing will be derived from training data that is wrong-by-construction for the new routing. Operators asking the model "how is claude -p billed?" after June 15 will receive answers presented with the same confident framing — but with even larger error.

4 · The 5-step pre-flight protocol

Total operator time: ~15 minutes (8 of which is the mandatory pause for the dashboard to update).

1

Baseline the dashboard

Before any test invocation, capture the baseline state of both relevant balances at the same wall-clock instant:

  • Max Plan / subscription quota usage: claude.ai/settings/usage
  • API account balance: console.anthropic.com/settings/billing

Note both numbers (or screenshots) with a timestamp. If your account doesn't use one of the two (e.g., Max Plan only, no API balance), record n/a for that field. The point is a before reading for both surfaces, so you can read the delta on each independently after the test invocation.

2

Run one isolated test invocation

From the same shell environment your automation will use (same OAuth or ANTHROPIC_API_KEY auth path, same CWD, same env vars):

claude -p "Respond with exactly: ok" 2>&1 | tee /tmp/claude-billing-preflight-$(date +%Y%m%d-%H%M%S).log

Record the exact timestamp. Token cost for a one-word response is roughly $0.01–$0.05 depending on input padding. The small magnitude is intentional: any non-zero charge against either bucket is unambiguously attributable to this test.

3

Wait 8 minutes

The billing dashboard does not update in real time. The lag is typically 3–6 minutes, occasionally longer. Wait 8 minutes minimum before reading the dashboard. Do not run a second test invocation during the wait — it will entangle the readings.

4

Read both dashboards again, compute deltas

After the pause, re-read both surfaces from step 1. Compute the delta on each:

  • Max Plan quota delta: units consumed since the baseline reading.
  • API balance delta: dollars charged since the baseline reading.

Exactly one of the deltas should be non-zero (modulo small measurement noise from other concurrent sessions, which is why this protocol assumes you can isolate the test from other automation during the 8-minute window). The non-zero delta identifies which bucket your test invocation charged. This is your ground truth — not the model's claims, not the docs, not your assumption.

5

Record the verification

Write a marker file recording the verified routing, the timestamp, and the auth path used. Re-run the protocol after any of these events:

  • Auth path change (switching between OAuth and API key)
  • Account-level billing changes (upgrade/downgrade Max Plan, change API balance limits)
  • Anthropic-side billing model changes (next known: 2026-06-15)
  • 30 days elapsed since last verification

5 · Reference script (MIT-licensed)

Save this as claude-print-billing-preflight.sh, chmod +x it, and run once per environment per auth path. The script does not read the dashboard automatically — Anthropic does not expose a public balance-delta API to non-organization operators as of 2026-05-24. The operator reads the dashboard manually after the 8-minute pause and types the deltas into the script's prompts.

#!/usr/bin/env bash
# claude-print-billing-preflight.sh
# Verify which credit bucket `claude -p` invocations charge against,
# before scaling automation.

set -euo pipefail

MARKER_DIR="${HOME}/.config/claude-billing-preflight"
LOG_DIR="/tmp"
TODAY=$(date +%Y-%m-%d)
TIMESTAMP=$(date -Iseconds)
TEST_LOG="${LOG_DIR}/claude-billing-preflight-$(date +%Y%m%d-%H%M%S).log"

mkdir -p "${MARKER_DIR}"

if ! command -v claude >/dev/null 2>&1; then
    echo "ERROR: claude command not found in PATH" >&2
    exit 1
fi

echo "=== claude -p billing pre-flight ==="
echo
read -rp "Press ENTER to read baseline dashboards, or Ctrl-C to abort... " _

echo "Baseline reading — open both URLs in a browser, record the numbers:"
echo "  Max Plan / subscription quota:  https://claude.ai/settings/usage"
echo "  API account balance:            https://console.anthropic.com/settings/billing"
read -rp "Max Plan quota baseline (or 'n/a'): " MAX_PLAN_BASELINE
read -rp "API balance baseline (or 'n/a'): " API_BALANCE_BASELINE

echo "Running test invocation..."
claude -p "Respond with exactly: ok" 2>&1 | tee "${TEST_LOG}" || {
    echo "ERROR: claude -p invocation failed. See ${TEST_LOG}" >&2
    exit 1
}

INVOCATION_TIME=$(date -Iseconds)

if [[ "${1:-}" != "--skip-pause" ]]; then
    echo "Pausing 8 minutes for dashboard to update..."
    sleep 480
fi

echo "Re-read both dashboards and compute deltas from the baseline:"
echo "  Max Plan / subscription quota:  https://claude.ai/settings/usage"
echo "  API account balance:            https://console.anthropic.com/settings/billing"
read -rp "Max Plan quota delta (or 'n/a' or 'zero'): " MAX_PLAN_DELTA
read -rp "API balance delta (or 'n/a' or 'zero'): " API_BALANCE_DELTA

echo "Which bucket showed the non-zero delta?"
echo "  1) max_plan_quota"
echo "  2) api_balance"
echo "  3) both (unusual — please document)"
echo "  4) neither (charge not yet reflected; re-run dashboard read in 5 minutes)"
read -rp "Enter 1-4: " ROUTING_CHOICE

case "${ROUTING_CHOICE}" in
    1) ROUTING="max_plan_quota" ;;
    2) ROUTING="api_balance" ;;
    3) ROUTING="both_unusual" ;;
    4) ROUTING="not_yet_reflected" ;;
    *) ROUTING="unknown" ;;
esac

MARKER_FILE="${MARKER_DIR}/${TODAY}.json"
cat > "${MARKER_FILE}" <<EOF
{
  "verified_at": "${TIMESTAMP}",
  "invocation_time": "${INVOCATION_TIME}",
  "test_invocation_log": "${TEST_LOG}",
  "max_plan_baseline": "${MAX_PLAN_BASELINE}",
  "max_plan_delta": "${MAX_PLAN_DELTA}",
  "api_balance_baseline": "${API_BALANCE_BASELINE}",
  "api_balance_delta": "${API_BALANCE_DELTA}",
  "routing_verified": "${ROUTING}",
  "auth_path_hint": "$([ -n "${ANTHROPIC_API_KEY:-}" ] && echo 'api_key_env' || echo 'oauth_or_other')",
  "claude_version": "$(claude --version 2>/dev/null || echo 'unknown')"
}
EOF

echo "=== Marker written: ${MARKER_FILE} ==="
cat "${MARKER_FILE}"

6 · Wrapper integration

To block automated claude -p invocations unless a recent marker file exists, wrap them with this check:

#!/usr/bin/env bash
# claude-p-wrapper.sh — block automated claude -p unless pre-flight is recent.

MARKER_DIR="${HOME}/.config/claude-billing-preflight"
MAX_AGE_DAYS=30

LATEST_MARKER=$(find "${MARKER_DIR}" -name '*.json' -type f -printf '%T@ %p\n' 2>/dev/null \
    | sort -rn | head -1 | cut -d' ' -f2-)

if [[ -z "${LATEST_MARKER}" ]]; then
    echo "ERROR: No claude-billing-preflight marker found. Run preflight first:" >&2
    echo "  ./claude-print-billing-preflight.sh" >&2
    exit 1
fi

MARKER_AGE_SECONDS=$(( $(date +%s) - $(stat -c %Y "${LATEST_MARKER}") ))
MAX_AGE_SECONDS=$(( MAX_AGE_DAYS * 86400 ))

if (( MARKER_AGE_SECONDS > MAX_AGE_SECONDS )); then
    echo "ERROR: claude-billing-preflight marker is older than ${MAX_AGE_DAYS} days." >&2
    echo "  Latest: ${LATEST_MARKER}" >&2
    echo "  Re-run: ./claude-print-billing-preflight.sh" >&2
    exit 1
fi

exec claude -p "$@"

Point your automation at ./claude-p-wrapper.sh instead of claude -p directly. The wrapper will refuse to run until the operator has pre-flighted within the last 30 days. On 2026-06-16 (the day after the cliff lands), every marker file will be invalidated in practice — even ones within the 30-day window — because the routing changed underneath them. Re-run the protocol on June 16 for every environment that runs -p automation.

7 · Integration by automation pattern

Bot pools (the #61704 jimpreuss-class case)

If you run a pool of N concurrent claude -p invocations:

  1. Pre-flight once for the auth path the pool uses (typically OAuth via spawn_bot.py or similar).
  2. Route the pool's invocations through claude-p-wrapper.sh rather than calling claude -p directly.
  3. If the pool scales up after the marker is invalidated (auth change, 30 days, June 15), the wrapper will block until re-verification.
  4. For pools that run continuously, schedule a daily summary that checks the marker age and emits a warning ≥7 days before expiry.

CI/CD (GitHub Actions, GitLab CI, Jenkins)

CI environments are ephemeral — the marker file does not persist between runs unless you cache it. Two patterns work:

  • Cached marker: store the marker file in CI cache (GitHub Actions cache action, GitLab CI cache stanza). The first CI run after a cache miss re-runs pre-flight; subsequent runs reuse the cached marker until expiry.
  • External marker check: store the marker in a secrets manager (AWS Secrets Manager, HashiCorp Vault, etc.) and fetch at CI start. This decouples marker age from CI runner lifecycle and is the recommended pattern for production automation.

Note that CI runners often use a separate auth path (API key in a secrets manager) than local development (OAuth). Pre-flight the CI auth path separately from your local auth path.

Scheduled jobs (cron, systemd timers)

Scheduled jobs running unattended are the highest-risk pattern: the operator is not present when the billing surprise occurs. Use the wrapper pattern with a hard floor on marker age (e.g., 14 days instead of 30), and send the operator a Slack/email alert 7 days before marker expiry.

Autonomous loops

Self-driving loops (autonomous agent systems, looping claude -p calls without human review) carry the same exposure shape as bot pools but with no concurrent operator attention to catch the failure. Apply the wrapper pattern with the tightest marker-age policy (14 days), and additionally log every invocation's token count to a local audit log that can be correlated against the dashboard manually.

8 · Post-cliff verification (June 16 onward)

The 2026-06-15 reclassification means every operator running -p automation needs to re-run the pre-flight protocol on or after June 16. The model's pre-cliff training data will be wrong-by-construction for the new routing; the dashboard reading is the only verification step that survives the routing change.

Calendar reminder: re-run pre-flight on 2026-06-16 (one day after the reclassification lands, to allow Anthropic's docs and the dashboard to fully reflect the new routing). Add this to your team calendar now — don't rely on remembering it after June 15.

Specific post-cliff sanity checks to add to your verification:

  1. The new SDK credit ceiling. Read the per-plan credit ceiling from support article 15036540 and confirm the dashboard surfaces it.
  2. The extra-usage flag. Verify whether your account has extra-usage enabled. If yes, post-cliff invocations beyond the credit will bill at API rates; if no, they will stop. Both outcomes have implications for automation that runs unattended.
  3. The auth path's bucket. Re-run the protocol from step 4 above and confirm the delta lands in the SDK-credit bucket (new) rather than the subscription-quota bucket (pre-cliff).

9 · Verification checklist before scaling

Before committing N concurrent claude -p invocations to a billing assumption, confirm:

  • ☐ I have personally verified the routing against the dashboard within the last 30 days (or 14 days if scheduled/autonomous).
  • ☐ The verification used the same auth path the scaled automation will use.
  • ☐ A marker file recording the verification exists at a stable path (not a temporary one).
  • ☐ The automation runs through a wrapper that checks marker age.
  • ☐ I have a Slack/email alert ≥7 days before marker expiry.
  • ☐ I have not relied on any model claim about billing as the basis for the assumption.
  • ☐ I have read support article 15036540 and understand the post-June-15 routing for my plan tier.
  • ☐ June 16 is on my calendar with a "re-run pre-flight" task.

If any line above is unchecked, the automation is exposed to the #61704 failure mode.
