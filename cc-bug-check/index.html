The 5 questions

1. Are you on Claude Code 2.1.143 (or later in the 2.1.x line)?

Check with claude --version in your shell. The find/grep nested-agent bug (issue #60325) was first reproduced on 2.1.143.

2. Do you have a ~/.claude/CLAUDE.md (or per-project CLAUDE.md) with explicit "don't do X" rules?

Examples: "No filler acknowledgments," "Never run rm -rf," "Don't push to main without confirmation." If the model has ever violated one of these mid-session despite the rule being loaded, you're hitting #60339.

3. Does your CLAUDE.md prohibit using TaskCreate / TaskList (because you use a separate task system)?

The harness injects a "consider TaskCreate" system reminder periodically. If your CLAUDE.md says don't use it, you're hitting #60323 — the reminder fires regardless of your opt-out.

4. Does your CLAUDE.md define a mandatory session-start protocol (e.g. "at session start, create file X, read directory Y")?

If the protocol is sometimes silently skipped without any error or warning, you're hitting #60337. Common shape: the model proceeds with the task as if the protocol completed.

5. Do you use Claude Code for security work (CVE repros, exploit drafting, pen-testing) where command accuracy is critical?

If the model has ever fabricated a generic command name (e.g. ./exploit) instead of the actual command established in the session (e.g. ./chage_pwn root), you're hitting #60340. Especially dangerous for repro hand-off documentation.
