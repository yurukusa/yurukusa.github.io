CLAUDE.md Recognition Check

A static analysis of your project CLAUDE.md against 8 patterns that are documented to be reachable-but-non-gating in the public GitHub issue tracker.

The thesis. "Write a stronger CLAUDE.md" is the mainstream remedy for Claude Code behaviour problems. Across 11+ public GitHub issues filed against anthropics/claude-code in 2026-04 to 2026-05, the operators who wrote the strongest possible rules — imperative, specific, in active context — saw the rules violated anyway. The failure mode is not weak rules; it is rules that the model can read, cite, and articulate while shipping the rule-violating action in the same response. This tool flags rule shapes whose strongest written form is documented to not gate.

What the tool checks

Each check looks for a rule shape that is documented to be in the recognition-without-arrest failure family. The check firing does not mean your rule is wrong; it means the rule strongest form is documented to not gate the action it tries to gate. The defense column is what actually works.

  • Path-exact directives — gated only at the model discretion. See #60068.
  • Anti-fabrication rules — emitted-as-correct claims ship anyway. See #56906.
  • Scope-creep prohibitions — additional files created anyway. See #60068.
  • STANDING RULE / NEVER / 絶対 markers — loaded into context, violated mid-session. See #59944.
  • Ask one clarifying question — questions skipped under ambiguity. See #60068.
  • Vendor-patch prohibitions — proposals violating the rule shipped despite memory in context. See #60226.
  • Verify-before-claim directives — "fix is deployed" claims ship without verification. See #60210.
  • Anti-completion-claim directives — "completed" emitted without verification tool calls. See #60177.

What the tool does not claim

Your rules might still help in some cases. The tool does not say your rules are useless; it says these specific rule shapes have documented failure modes where the strongest-possible-articulation does not gate. The right response to a finding is to add a runtime-side defense, not to remove the rule.

The defenses that work

The runtime-side hooks in cc-safe-setup (730+ MIT-licensed PreToolUse and PostToolUse hooks) include several that address this failure family directly: rhetorical-verification-prompt-detector (input side), completion-claim-without-verification-detector (output side), denyread-fallback.sh (#58636), and the SessionStart standing-rule surfacer (#59944).
